## En une phrase

**Estime Vangu** est un **ingénieur frontend orienté produit** : il conçoit et livre des applications web modernes (SaaS, outils métier, sites premium) avec une forte attention à l’expérience utilisateur, à la clarté produit et à la qualité d’exécution.

Ce site est sa **vitrine professionnelle** : pas une simple page de présentation, mais un produit web à part entière — structuré, traduit, animé avec sobriété, et pensé pour être lu comme on lirait un dossier de compétences premium.

## Ce que le portfolio montre (et ce qu’il ne montre pas)

### Ce qu’il montre

- Une **sélection** de projets — qualité et cohérence plutôt que volume
- La capacité à livrer des **produits complets** : apps métier, plateformes, sites vitrine
- Une **réflexion produit** : problème, approche, décisions, exécution
- Un **niveau frontend élevé** : navigation fluide, mobile-first, animations maîtrisées
- Un **parcours contact** adapté à votre type de projet

### Ce qu’il ne montre pas

- Un inventaire exhaustif de tous les projets réalisés
- Un catalogue de technologies sans contexte métier
- Un site « template » générique type portfolio créatif standard
- Le code source sur le site lui-même (le dépôt est public séparément pour les profils techniques)

## Fonctionnalités globales

### Langues

Le site est disponible en **français** et en **anglais**. Le sélecteur de langue se trouve dans l’en-tête (icône globe). Le contenu — pages, projets, formulaire — suit la langue choisie.

### Thème clair / sombre

Un bouton permet de basculer entre **mode clair** et **mode sombre**. La transition est animée ; si vous préférez réduire les animations (réglage système « réduire les animations »), le site respecte ce choix.

### Navigation

| Appareil       | Comportement                                                                   |
| -------------- | ------------------------------------------------------------------------------ |
| **Ordinateur** | Menu horizontal centré : Accueil, Profil, Projets, Contact                     |
| **Mobile**     | Menu via le bouton en haut à droite ; panneau plein écran avec les mêmes liens |

Sur les **pages d’étude de cas** (détail d’un projet), la navigation principale est volontairement masquée : seul un **bouton retour** vers la liste des projets reste visible, pour garder le focus sur la lecture.

### Accessibilité

- Lien « Aller au contenu principal » pour sauter l’en-tête
- Navigation au clavier et libellés pour les lecteurs d’écran
- Contraste et lisibilité pensés pour les deux thèmes

### Bandeau contact rapide

Sur l’**accueil**, des raccourcis permettent de joindre Estime directement :

- LinkedIn
- Instagram
- E-mail
- Téléphone

## Parcours page par page

### 1. Entrée — Splash (`/`)

Courte animation de marque à l’ouverture du site, puis redirection vers l’accueil. C’est l’« entrée en matière » du portfolio — sobre, premium.

**Rôle :** positionnement en quelques secondes.

### 3. Profil (`/profile`)

**Rôle :** expliquer _comment_ Estime travaille — pas une biographie classique.

Six sections thématiques avec sommaire latéral (ordinateur) ou horizontal (mobile) :

| Section                   | Contenu                                              |
| ------------------------- | ---------------------------------------------------- |
| Vision produit            | MVP, arbitrages, pas de features inutiles            |
| Philosophie frontend      | Fluidité, mobile, cohérence visuelle                 |
| Ce que je construis       | SaaS, outils métier, dashboards, apps denses         |

**Rôle :** montrer des **preuves** — projets sélectionnés avec contexte.

1. **Études de cas** — 4 projets détaillés (voir section suivante)

**Rôle :** comprendre un projet en profondeur — problème, approche, architecture, décisions.

Réservée aux 4 projets mis en avant. Structure détaillée plus bas.

### 6. Contact (`/contact`)

**Rôle :** démarrer une collaboration avec le bon niveau d’information.

Formulaire guidé par **type de besoin** + canaux directs. Voir section Contact.

## Les études de cas

Chaque étude de cas suit la **même trame** pour faciliter la lecture et la comparaison.

### Structure d’une étude de cas

| Bloc                   | Ce que vous y lisez                                             |
| ---------------------- | --------------------------------------------------------------- |
| **En-tête**            | Nom, type, année, statut, rôle d’Estime, tags (enjeux + stack)  |
| **Problème**           | Le besoin métier, les tensions, ce qui n’allait pas             |
| **Approche**           | Les hypothèses, les choix, les compromis assumés                |
| **Contexte produit**   | Où s’inscrit le projet, pour qui, quel périmètre                |
| **Architecture**       | Comment le produit est structuré (sans noyer dans le technique) |
| **Décisions clés**     | Arbitrages importants et pourquoi                               |
| **Exécution et suite** | Où en est le projet, prochaines étapes                          |

### Sommaire (table des matières)

Sur **ordinateur**, un sommaire fixe à gauche suit votre lecture :

- la section en cours est **soulignée**,
- le scroll vertical de la page met à jour l’indicateur.

Sur **mobile**, le sommaire défile horizontalement sous l’en-tête :

- l’entrée active se place en tête de la zone visible,
- en bas de page, les derniers items restent alignés proprement.

### Retour

Un bouton **← Retour aux projets** remplace le logo sur les pages d’étude de cas.

## La page Profil

La page Profil répond à la question : **« Est-ce que ce profil correspond à notre besoin ? »**

Plutôt qu’une liste de diplômes ou d’emplois, vous y lisez une **méthode de travail** :

- comprendre le besoin avant la solution,
- livrer des MVP ciblés,
- privilégier la clarté et la maintenabilité,
- soigner le frontend comme levier produit (pas comme décoration).

Le sommaire permet de sauter directement à la section qui vous intéresse — utile si vous recrutez pour du **SaaS**, du **B2B** ou des **outils internes**.

## Questions fréquentes

### Le site est-il un template ?

Non. C’est un produit sur mesure — architecture dédiée, contenu rédigé, animations et navigation pensées pour ce portfolio précis.

### Pourquoi seulement 4 études de cas ?

Pour garder une lecture profonde et crédible. Les autres projets sont listés avec lien live quand ils sont en production.

### Estime fait-il aussi du backend ?

Oui, de façon **mesurée** : auth, bases de données (souvent Supabase), APIs, validation — surtout quand le produit l’exige. Le positionnement reste **frontend-first** : l’expérience utilisateur et la structure produit guident les choix techniques.

### Peut-on voir le code ?

Le dépôt est **open source** sur GitHub (`portfolio_pro`) pour les profils techniques. Les projets clients listés ont leurs propres dépôts selon les accords.

### Le site est-il optimisé mobile ?

Oui — **mobile-first** : navigation, formulaire, sommaires et lisibilité sont pensés d’abord pour le téléphone.

### Les animations gênent-elles la lecture ?

Non — elles sont **sobres et utiles** (transitions de menu, thème, entrée). Le site respecte le réglage « réduire les animations » du système.

## Glossaire

| Terme                | Signification simple                                           |
| -------------------- | -------------------------------------------------------------- |
| **Étude de cas**     | Page détaillée sur un projet : problème, solution, décisions   |
| **MVP**              | Version minimale utilisable, pour tester en conditions réelles |
| **PWA**              | Site installable sur téléphone comme une application           |
| **SaaS**             | Logiciel en ligne par abonnement (ex. outil métier web)        |
| **B2B**              | Produit destiné aux entreprises, pas au grand public           |
| **i18n**             | Site disponible en plusieurs langues                           |
| **Frontend**         | Partie visible et interactive du site / de l’app               |
| **Fullstack mesuré** | Capacité backend quand nécessaire, sans sur-promesse           |
