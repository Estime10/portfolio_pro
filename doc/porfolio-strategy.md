# Portfolio Rebuild — Strategy & Phases

## Vision

Créer un portfolio moderne, premium et technique qui reflète un vrai profil de product engineer.

Le portfolio ne doit pas ressembler à :

- une simple landing page de développeur
- un portfolio Awwwards générique
- une grille de projets avec animations inutiles

Le but est de construire une expérience qui démontre :

- excellence **frontend** et exécution UI
- capacité à livrer des **produits complets** (fullstack mesuré, sans posture excessive)
- réflexion **produit** et sens **business**
- qualité d’exécution
- architecture moderne
- sens du détail
- capacité à **ship** des applications réelles

---

# Core Identity

## Positionnement

**Frontend orienté produit**, avec de fortes capacités fullstack — le terme « fullstack » reste **mesuré**, sans posture « expert de tout le backend ».

Formulations cibles :

- frontend engineer
- product-oriented developer
- modern web application builder
- profil **frontend-first fullstack**

Le portfolio doit transmettre la capacité à livrer des **produits complets**, une **architecture moderne**, une **logique produit / business**, une **vision UX**, et une **exécution frontend excellente**.

> **Détail Phase 1 (audience, ton, forces, perception)** : voir [`phase-1-identity.md`](./phase-1-identity.md).

Le portfolio doit transmettre l’idée suivante :

> « Ce développeur sait construire des applications modernes sérieuses avec une vraie réflexion produit et une excellente exécution frontend. »

# Portfolio Architecture

## Structure hybride

Le portfolio ne sera probablement pas une simple one-page.

Structure envisagée :

### Home

Page immersive et narrative.

Objectifs :

- créer une identité
- montrer le niveau frontend
- introduire les projets
- présenter l’approche engineering

---

### Work

Vue projets / systems.

Pas une simple grille.

Pistes :

- timeline interactive
- navigation cinématique
- cards techniques
- système de filtres
- navigation fluide

---

### Case Studies

Pages dédiées aux projets.

Exemple :

- FleetScan

Ces pages doivent montrer :

- contexte
- problèmes
- architecture
- décisions techniques
- contraintes
- UX
- logique produit
- roadmap
- vision
- apprentissages

Objectif :
montrer la pensée engineering et pas uniquement le résultat visuel.
